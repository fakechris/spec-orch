# Task Contract: T3.3 — GateService.evaluate 实现 promotion_required 逻辑

> Change: 01-scaffold-flow-engine
> Task ID: T3.3
> 前置: T3.1 (GateInput 扩展), T1.3 (FlowTransitionEvent + GateVerdict 扩展)
> Spec 对应: REQ-3 (Gate 触发的流程升级)

---

## 1. Intent

在 `GateService.evaluate()` 中增加一个判断：如果 issue **声称是低流程**
（如 Hotfix 或 doc-only Standard），但 **实际改动不符**（如改了代码文件），
则在返回的 `GateVerdict` 中标记 `promotion_required=True` 并指明
`promotion_target`。

**外部可观测结果**：调用 `GateService.evaluate(gate_input)` 后，返回值
`GateVerdict` 多出两个字段，调用方可据此决定是否升级流程。
Gate 本身不执行升级，只产出信号。

---

## 2. Decisions

### 已确定

| 决定 | 理由 |
|------|------|
| `promotion_required` 和 `promotion_target` 作为 GateVerdict 的**可选字段**，默认 False / None | 不破坏现有消费方（RunController、CLI、Daemon 都读 GateVerdict） |
| 首版只实现一条规则：`claimed_flow` 为 Hotfix 或 Standard 且 `diff_stats` 包含非文档文件 → promotion_required | 简化版够用，更多规则留给 GatePolicyEvolver |
| `diff_stats` 作为 GateInput 的新字段传入，类型为 `dict[str, int]` （如 `{"py": 3, "md": 1}`）| Gate 不应自己跑 git diff；调用方（RunController）负责收集 |
| `claimed_flow` 作为 GateInput 的新字段，类型为 `str | None`（如 `"hotfix"`, `"standard"`）| Gate 需要知道 issue 声称的流程才能判断不一致 |
| promotion 判定逻辑放在 `evaluate()` 内部，紧跟现有条件检查之后 | 保持单一入口，不拆新方法 |
| promotion 判定**不影响 mergeable**——即使 promotion_required=True，如果所有条件都过，mergeable 仍为 True | promotion 是信号，不是 gate fail；由 RunController 决定是否响应 |

### Open Questions

| 问题 | 影响 | 建议 |
|------|------|------|
| "非文档文件"的判定标准是什么？是后缀白名单（`.md`, `.txt`, `.rst`）还是路径前缀（`docs/`）？ | 影响 promotion 触发的精确度 | 建议首版用后缀白名单 `{".md", ".txt", ".rst", ".json"}`，可配置 |
| promotion_target 由 Gate 决定还是由配置决定？ | 影响 Gate 是否需要读 flow_mapping | 建议首版硬编码：Hotfix→Standard, Standard(doc-only)→Standard(full), Standard(code)→Full |

## 3. Boundaries

### Allowed Paths

- 修改 `src/spec_orch/domain/models.py`：
  - `GateInput` dataclass 增加 `claimed_flow: str | None = None` 和
    `diff_stats: dict[str, int] = field(default_factory=dict)`
  - `GateVerdict` dataclass 增加 `promotion_required: bool = False` 和
    `promotion_target: str | None = None`
- 修改 `src/spec_orch/services/gate_service.py`：
  - `GateService.evaluate()` 方法内部，在现有条件检查之后增加 promotion 判定逻辑
- 新增测试文件或在 `tests/unit/test_gate_service.py` 中增加测试

### Forbidden Paths

- **不得修改** `GateService.__init__`, `GatePolicy`, `GatePolicy.from_yaml`,
  `GatePolicy.with_profile`, `should_auto_merge`, `describe_policy`, `describe_as_dict`
- **不得修改** `gate.policy.yaml`
- **不得修改** `RunController`（T4.5 的职责）
- **不得引入** FlowType 枚举或 FlowGraph——这些是 T1.1/T1.2 的职责；
  本 task 中 `claimed_flow` 和 `promotion_target` 都是 `str`，不是枚举
- **不得导入** `conductor` 模块——Gate 不知道 Conductor 的存在
- **不得修改** 现有 GateInput 字段的默认值或语义
- **不得在 evaluate() 中调用外部服务**（不调 git、不调 LLM、不调 Memory）

### Public Behavior That Must Not Regress

| 行为 | 验证方式 |
|------|---------|
| `evaluate(GateInput())` 返回 `mergeable=False`，`promotion_required=False` | 现有 test_default_policy_blocks_all_by_default |
| 全量 pass 时 `mergeable=True`，`promotion_required=False` | 现有 test_custom_policy_removes_human_acceptance |
| profile 机制、auto_merge 机制不受影响 | 现有 test_gate_strategy.py 全部 14 个测试 |
| `GateVerdict` 可以不传新字段构造（默认值兜底）| 现有 test_gate_verdict_fields |
| `_build_gate_input_from_report` 不传新字段时不报错 | 现有 test_build_gate_input_from_report |

## 6. Risk Notes

### 最可能越界的点

| 风险 | 描述 | 防护 |
|------|------|------|
| **引入 FlowType 枚举** | agent 可能把 `claimed_flow: str` 改为 `FlowType` 枚举引用，但 FlowType 还不存在（T1.1 的产出） | contract 明确禁止：claimed_flow 和 promotion_target 都是 `str`，不是枚举 |
| **在 evaluate 里跑 git diff** | agent 可能认为应该在 Gate 内部收集 diff_stats | contract 明确禁止：diff_stats 由调用方传入，Gate 不调外部命令 |
| **修改 mergeable 语义** | agent 可能认为 promotion_required 应该让 mergeable=False | contract 明确：C6 要求 promotion 不影响 mergeable |
| **过度重构 evaluate()** | 现有 evaluate() 是简单的 if-chain，agent 可能重构为循环或策略模式 | contract 明确：不得修改现有条件检查逻辑的结构，只在末尾追加 |

### 最可能误改的模块

| 模块 | 原因 | 防护 |
|------|------|------|
| `run_controller.py` | 自然想"升级流程就要改 RunController" | Forbidden Paths 明确排除 |
| `gate.policy.yaml` | 可能想把 promotion 作为可配置条件加到 YAML | Forbidden Paths 明确排除 |
| `cli.py` `_build_gate_input_from_report` | 可能想给这个函数也加 diff_stats 解析 | 不在 Allowed Paths 中；且此函数从 report.json 构建，diff_stats 需要另一个数据源 |
