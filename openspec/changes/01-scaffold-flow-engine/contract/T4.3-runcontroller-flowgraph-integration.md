# Task Contract: T4.3 — RunController 按 FlowGraph 选择初始步骤

> Change: 01-scaffold-flow-engine
> Task ID: T4.3
> 前置: T4.2 (_resolve_flow), T2.1 (FlowEngine.get_graph)
> Spec 对应: REQ-8.1, REQ-8.2, S1.1, S1.2, S1.3

---

## 1. Intent

修改 `RunController.run_issue()` 使其根据已解析的 `FlowType` 从 `FlowEngine` 获取
`FlowGraph`，并以图的第一个步骤作为初始步骤，而非依赖硬编码的隐式流程。

**外部可观测结果**：
- Full 流程的 `run_issue` 从 `discuss` 步骤开始
- Standard 流程从 `implement` 步骤开始
- 未指定时默认走 Standard，行为与当前等价

---

## 2. Decisions

### 已确定

| 决定 | 理由 |
|------|------|
| `FlowEngine` 和 `FlowMapper` 通过 `__init__` 注入 RunController，均有默认实例 | 避免破坏现有构造方式（REQ-8.2） |
| `run_issue(issue_id, flow_type=None)` 签名扩展为可选参数 | 现有调用方不传 flow_type 时走 `_resolve_flow` |
| `current_step` 和 `current_flow` 作为 `run_issue` 内部局部变量维护 | 不增加 RunController 实例状态；每次 run 独立 |
| FlowStep 与 RunState 的映射关系定义在 FlowGraph 中 | RunController 通过 `step.run_state` 判断当前应执行什么操作 |
| 初始步骤逻辑仅影响 `run_issue`，不影响 `advance` 和 `advance_to_completion` | 渐进式改造，先改 run_issue |

### Open Questions

| 问题 | 影响 | 建议 |
|------|------|------|
| `advance()` 是否也需要改为按 FlowGraph 驱动？ | 影响 T4.4 的范围 | 本 task 不改 advance，T4.4 负责 |
| Full 流程的步骤中有些（如 discuss, freeze）无对应 RunState，如何处理？ | 影响流程推进逻辑 | 这些步骤标记为 `run_state=None`，run_issue 跳过无 run_state 的步骤（等 T4.4 完善） |

## 3. Boundaries

### Allowed Paths

- 修改 `src/spec_orch/services/run_controller.py`：
  - `__init__` 增加 `flow_engine` 和 `flow_mapper` 可选参数
  - `run_issue` 签名增加 `flow_type: FlowType | None = None`
  - `run_issue` 内部调用 `FlowEngine.get_graph(flow_type)` 获取初始步骤
- 新增或修改 `tests/unit/test_run_controller.py` 和 `tests/integration/` 中的相关测试

### Forbidden Paths

- **不得修改** `FlowEngine`, `FlowGraph`, `FlowMapper` 的实现（T2.x 的产出）
- **不得修改** `GateService` 的任何代码
- **不得修改** `advance()` 或 `advance_to_completion()` 的逻辑（T4.4 的职责）
- **不得修改** `domain/models.py` 中已有的字段定义
- **不得引入** 新的依赖库
- **不得改变** `RunResult` 的返回结构
- **不得删除** RunController 现有的任何公开方法

### Public Behavior That Must Not Regress

| 行为 | 验证方式 |
|------|---------|
| `run_issue(issue_id)` 不传 flow_type 时返回与当前完全一致的 RunResult | 现有集成测试 |
| `advance(issue_id)` 行为不变 | 现有测试 |
| `advance_to_completion(issue_id)` 行为不变 | 现有测试 |
| GateInput/GateVerdict 的构造方式不受影响 | 现有 gate 测试 |
| `pytest --tb=short` 全量通过 | 全量回归 |

## 6. Risk Notes

### 最可能越界的点

| 风险 | 描述 | 防护 |
|------|------|------|
| 重构 run_issue 全部逻辑 | 容易在改初始步骤时顺手重构整个 run_issue | contract 限定仅改初始步骤选择，不改 advance 逻辑 |
| 修改 advance 方法 | 改了 run_issue 自然想把 advance 也改成图驱动 | Forbidden Paths 明确排除 |
| 为 Full 流程实现完整的 discuss/freeze 步骤逻辑 | 初始步骤选到 discuss 后想实现该步骤的完整功能 | 本 task 只负责"选择初始步骤"，步骤内逻辑是后续 task |

### 最可能误改的模块

| 模块 | 原因 | 防护 |
|------|------|------|
| `gate_service.py` | RunController 调用 Gate 时想传更多参数 | Forbidden Paths |
| `domain/models.py` | 想给 RunResult 加 flow_type 字段 | Forbidden Paths |
| `flow_engine/` | 想调整 FlowGraph 定义来配合 RunController | Forbidden Paths，这些是 T2.x 的产出 |
