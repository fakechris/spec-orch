# SpecOrch

**Spec 驱动的软件交付控制面。**

> [English Version](VISION.md)

---

SpecOrch 不是一个帮你写代码的聊天工具。
不是又一个泛化的多智能体框架。
不是一个想替代 IDE 的编辑器。

**SpecOrch 是一个软件交付控制面。**

它把原本散落在 issue 跟踪、spec 编写、agent 执行、代码审查、合并门控中的交付过程，统一到一个以 **Spec 为锚点**、以 **Evidence 为证明**、以 **Gate 为完成态**、以 **Evolution 为持续改进** 的系统中。

我们的目标不是让 Agent 看起来更聪明。
我们的目标是让软件交付变得 **可编排、可验证、可运营、可演化**。

---

## 为什么需要 SpecOrch

今天大多数 AI 开发工具解决的是"如何让模型多写一点代码"。但真正的软件交付困难从来不只是代码生成：

- **需求不断漂移**，缺少冻结后的权威定义
- **Agent 执行不稳定**，不可重复、不可追溯
- **PR 合并了**，却没人真正知道是否满足了原始目标
- **Review 盯着 diff**，看不到行为、风险、偏离和证据
- **Prompt 越写越长**，系统越来越脆，最后谁都说不清它为什么成功

这些问题不能靠再加一个 Agent 解决。它们需要更高一层的系统——一个能把 **意图、任务、执行、验证、进化** 串起来的控制面。

**这就是 SpecOrch 的位置。**

## 核心理念

### 1. Spec 才是需求

Issue 不是需求。聊天记录不是需求。PR 也不是需求。

**冻结后的 Spec，是这次交付中的唯一真相。**

SpecOrch 的一切执行、验证、审查和演化，都以 Spec 为锚点。没有锚点的 Agent，只是高成本的即兴发挥。

### 2. Merge 不是完成态——Gate 才是

代码生成了不代表工作完成。PR 合并了也不代表目标达成。

**完成态由证据定义，不由活动定义。**

在 SpecOrch 里，真正的"完成"来自 Gate：验收标准满足、验证通过、偏差已解释、风险已评估、证据充分。

### 3. 先做 Orchestration，再考虑 Multi-Agent

行业热炒 multi-agent，但绝大多数真实工作流本质上仍是编排：

- 一个主系统做决策
- 多个 worker 执行独立任务
- 任务通过结构化输入输出衔接
- 上层统一汇总、验证、决策

**SpecOrch 默认 orchestration-first。** 只有当任务依赖、角色协商、共享状态真正成为瓶颈时，才升级到更丰富的协调模式。

### 4. Prompt 是建议——Harness 是强制

Prompt 很重要，但 Prompt 不是系统。

如果一个流程必须被稳定执行，它就不该只存在于 Prompt 里。它必须沉淀成脚本、策略、门控、钩子、反应规则和运行时约束。

**SpecOrch 的核心不是"怎么写更好的 Prompt"，而是"怎么把关键执行逻辑压成系统轨道"。**

### 5. Context 是可治理的契约

上下文不是一大坨提示词。它应该被设计、被约束、被分层装配：

- **项目级**：长期稳定的规则和约定
- **Spec 级**：本次交付的边界与目标
- **任务级**：具体工作包的输入输出
- **会话级**：执行过程中的临时状态
- **消息级**：当前轮次的即时补充

没有上下文治理，Agent 只能依赖偶然记忆。没有上下文边界，系统会越来越贵、越来越慢、越来越脆。

### 6. Evidence 是系统的记忆

系统真正的记忆不是聊天记录，而是结构化证据：决策、diff、发现、测试结果、偏差、门控结果、复盘、运行轨迹。

这些 artifacts 不是副产物，它们是系统能否学习、解释和自我修正的基础。

**没有 Evidence，进化就是玄学。有了 Evidence，进化才是工程。**

### 7. 进化 Harness，而不只是进化 Prompt

SpecOrch 不是 prompt 优化器，而是一个能学习的编排系统。

真正的优化对象包括：上下文装配、任务分解、技能加载、策略提取、审查结构、门控阈值、运行时反应、工具表面和执行配方。

Prompt 当然也会进化——但 Prompt 只是 Harness 中的一个旋钮。

## 七层架构（Seven Planes）

SpecOrch 将软件交付组织为七个平面：

### 契约平面 (Contract Plane)
*"我们到底在交付什么？"*

管理 spec、scope、验收标准、决策、约束和待解决的问题。Spec 在此冻结，成为唯一真相。

### 任务平面 (Task Plane)
*"为了完成 Spec，如何分解工作？"*

将 spec 展开为可执行的任务图，含依赖、所有权、阻塞、状态转换和工作包。支持波次并行执行。

### 装配平面 (Harness Plane)
*"如何让执行变稳定，而不是靠运气？"*

管理上下文契约、技能、策略、设置钩子、运行时规则、反应机制、回退策略和缓存安全的 prompt 布局。这是系统改进的主战场。

### 执行平面 (Execution Plane)
*"谁来做，在哪里做，以什么隔离边界来做？"*

管理 worktree、沙箱、会话、分支、运行时提供方和 agent 适配器。一任务一隔离——始终如此。

### 证据平面 (Evidence Plane)
*"凭什么相信这次交付是正确的？"*

管理发现、测试、预览链接、diff、偏差、审查摘要、风险信号和门控证据。完成是被证明的，不是被声称的。

### 控制平面 (Control Plane)
*"一个人如何运营一群任务和 Agent？"*

管理 mission、任务图、会话、PR 生命周期、门控状态、事件、成本、卡住的运行和通知。运营指挥中心。

### 进化平面 (Evolution Plane)
*"系统如何随时间变得更好？"*

管理运行轨迹、复盘、prompt 进化、策略蒸馏、装配评估和系统级调优。每次运行都反馈到下一次。

## SpecOrch 不是什么

- **不是聊天玩具**。它是一个有结构化契约和证据的交付系统。
- **不是多智能体社交实验**。默认 orchestrator-worker，不是对等协商。
- **不是 IDE 替代品**。有 dashboard 和 TUI，但核心身份是控制面。
- **不是 PR 机器人**。不只是把 issue 变成 PR，而是管理从 spec 到 evidence 到 evolution 的完整生命周期。

我们不追求表面的 agent 自主性。我们追求 **交付完整性（delivery integrity）**。

## 设计原则

| 原则 | 含义 |
|------|------|
| **Spec 优先** | 所有执行从冻结的 spec 出发，不从模糊意图出发 |
| **编排优先** | 默认清晰的中心控制，而非自发协商 |
| **Harness 优先** | 先固化执行轨道，再优化模型行为 |
| **证据优先** | 没有证据，就没有完成态 |
| **人类治理，机器执行** | 人类负责目标、边界、判断和批准；机器负责分解、执行、验证和汇报 |
| **骨架确定，智能可变** | 骨架必须稳定，智能可以进化，进化不能破坏骨架 |
| **渐进操作化** | 先写成 Skill → 蒸馏成 Policy → 固化成 Script/Gate/Hook。从经验走向机制 |
